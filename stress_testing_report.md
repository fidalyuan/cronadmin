# 📋 CronAdmin Web Application Stress Testing Report
## (CronAdmin 系统压力测试报告)

**测试时间**: 2026-06-23  
**测试目标**: `https://115.190.183.156:5173` (后端 API: `/api/v1`)  
**压测工具**: 基于 Python `ThreadPoolExecutor` 编写的定制化压测脚本（设置连接/读取超时为 5 秒，以模拟真实高并发下客户端的超时退出）  

---

## 1. 执行摘要 (Executive Summary)

本报告针对 CronAdmin 系统的核心接口在不同并发级别下（**10、30、50 并发线程**，每个测试周期发送 **150 个请求**）的响应性能、吞吐量（QPS）和稳定性进行了全面评估。

### 核心结论：
1. **系统轻载表现良好**: 在 10 个并发线程以下，静态配置接口及状态检测接口响应迅速，整体成功率接近 100%。
2. **严重的 CPU 瓶颈 (Bcrypt 密码哈希)**: 登录接口 `/auth/login` 在并发达到 30 和 50 时，由于后端采用 Bcrypt 强加密算法进行密码验证，导致服务器单核 CPU 瞬间被打满，大量请求堆积，最终 100% 超时失败。
3. **系统调用开销大 (`/ports/status`)**: 端口状态扫描接口在并发增加到 30 和 50 时，也出现了严重的超时问题，成功率降至 0%，反映出实时调用系统命令/扫描网络状态对资源的消耗极高。
4. **级联效应与连接池堆积**: 高压力测试后，服务器因处理积压任务导致后续的测试（如 `tasks_list` 10并发阶段）在开始时受到波及，产生级联超时。

## 3. 根因与瓶颈分析 (Bottleneck Analysis)

### 3.1 验证码与登录密码哈希 (Bcrypt) 的高能耗
Bcrypt 算法在设计上就是为了故意拖慢速度以抵御暴力破解（Work Factor）。
- **现象**: 在没有做限流的情况下，恶意或并发的登录请求会瞬间吃光服务器的 CPU 时间片。
- **影响**: 阻塞了整个 ASGI 异步应用（FastAPI / Uvicorn）的事件循环，导致其他轻量级请求（如任务列表、甚至静态配置）无法被调度执行，引发大面积超时。

### 3.2 动态系统调用接口的无缓存扫描
`/ports/status` 每次被动请求都会触发系统调用获取进程信息。
- **现象**: 这是一个典型的“拉取（Pull）型”低效设计，在高并发下每个请求都去调用系统命令。
- **影响**: 系统资源被无意义的重复进程状态扫描耗尽，造成高昂的 IO/CPU 上下文切换开销。

## 4. 优化改进建议 (Recommendations)

为了提高 CronAdmin 系统的稳定性和并发承受能力，建议针对性地采取以下优化措施：

### 🛠️ 建议一：对高危/高能耗接口引入限流 (Rate Limiting)
- **方案**: 引入 `slowapi` 或使用 Redis 等工具对 `/auth/login` 登录接口进行严格的并发和频率限制（例如：每个 IP 每分钟最多请求 10 次）。
- **目的**: 避免恶意刷接口或短时并发登录打爆服务器，保护系统核心功能。

### 🛠️ 建议二：引入数据缓存机制 (Caching) - 针对端口状态接口
- **方案**: 改变 `/ports/status` 的数据获取模式。**绝不在请求到达时才去实时扫描**。
  1. 后端使用一个轻量级的后台定时任务（如 `APScheduler`），每 5-10 秒扫描一次端口与进程状态，并将结果写入内存字典（全局变量）或 Redis 缓存中。
  2. `/ports/status` 接口收到请求后，**直接读取并返回缓存中的最新数据**。
- **目的**: 使该接口的查询复杂度由 $O(SystemScan)$ 降到 $O(1)$，即使在 1000 并发下也能在数毫秒内响应。

### 🛠️ 建议三：优化后端部署架构 (Scaling & Workers)
- **方案**: 
  - 目前容器中可能仅运行了单 Worker 的 Uvicorn 实例。在多核服务器上，建议使用 `--workers` 参数（如 `uvicorn main:app --workers 4`）来部署多个 ASGI 工作进程。
  - 对于极其耗费 CPU 的 Bcrypt 校验，可以考虑将其放入 `asyncio` 的线程池中运行（FastAPI 默认在遇到普通的 `def` 同步函数时会自动在线程池中处理，确保 `login` 路由没有被误声明为 `async def` 却包含耗时的同步哈希操作）。
